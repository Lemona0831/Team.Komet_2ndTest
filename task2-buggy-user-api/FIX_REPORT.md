# FIX_REPORT

## 1. 발견한 문제

- 문제 1: API 응답 상태 코드가 상황에 맞게 반환되지 않는 문제가 있었음. 사용자 생성, 삭제, 조회 실패 등 서로 다른 상황에서도 대부분 200 OK로 처리되어 클라이언트가 요청의 성공 여부를 명확히 판단하기 어려웠음.
- 문제 2: 존재하지 않는 사용자에 대한 예외 처리가 안전하지 않았음. Repository에서 `First()`를 사용하거나 조회 결과에 대한 null 처리가 부족하여, 존재하지 않는 id를 조회, 수정, 삭제할 때 서버 예외가 발생할 수 있었음.
- 문제 3: 사용자 삭제 처리가 불완전했음. 삭제 성공 여부를 확인하지 않고 항상 성공 응답을 반환하거나, 이미 삭제된 사용자와 존재하지 않는 사용자를 구분하지 못하는 문제가 있었음. 또한 삭제된 사용자가 조회 결과에 노출될 가능성이 있었음.
- 문제 4: 사용자 생성 및 권한 변경 시 입력값 검증이 부족했음. email, password, name이 비어 있어도 사용자가 생성될 수 있었고, role 값이 `USER`, `ADMIN`이 아니어도 저장될 수 있는 문제가 있었음.
- 문제 5: 사용자 데이터 응답 및 중복 검사에 문제가 있었음. 이메일 중복 검사가 없거나 대소문자를 구분하여 같은 이메일이 중복 등록될 수 있었고, 응답 DTO에 password가 포함되거나 name 필드에 실제 이름이 아닌 다른 값이 들어갈 가능성이 있었음.

## 2. 수정한 내용

- 수정 1: API 엔드포인트에서 상황에 맞는 HTTP 상태 코드를 반환하도록 수정함. 사용자 생성 성공 시 `201 Created`, 조회 성공 시 `200 OK`, 삭제 성공 시 `204 No Content`, 잘못된 요청은 `400 Bad Request`, 존재하지 않는 사용자는 `404 Not Found`, 이메일 중복은 `409 Conflict`를 반환하도록 변경하여 클라이언트가 요청 결과를 명확하게 판단할 수 있도록 개선함.
- 수정 2: 사용자 조회 로직에서 안전한 조회 방식을 사용하도록 수정함. 기존에는 조건에 맞는 사용자가 없을 때 예외가 발생할 수 있었으므로,`First()` 대신 `FirstOrDefault()`를 사용하여 조회 실패 시 null을 반환하도록 변경함. 이후 Service 또는 API 엔드포인트에서 null 여부를 확인하여 존재하지 않는 사용자에 대해 `404 Not Found`를 반환하도록 처리함.
- 수정 3: 사용자 삭제 로직을 삭제 성공 여부를 반환하는 방식으로 수정함. 존재하지 않는 사용자 또는 이미 삭제된 사용자를 삭제하려는 경우 `false`를 반환하고, API에서는 이를 `404 Not Found`로 처리하도록 변경함. 정상적으로 삭제된 경우에는 `IsDeleted` 값을 true로 변경하고 `204 No Content`를 반환하도록 수정함. 또한 삭제된 사용자는 목록 조회와 단건 조회 결과에서 제외되도록 개선함.
- 수정 4: 사용자 생성 및 권한 변경 시 입력값 검증을 추가함. 사용자 생성 시 `email`, `password`, `name`이 비어 있는 경우 `400 Bad Request`를 반환하도록 처리하고, role 값은 `USER`, `ADMIN`만 허용하도록 검증함. 권한 변경 요청에서도 잘못된 role 값이 들어오면 `400 Bad Request`를 반환하도록 수정함.
- 수정 5: 사용자 응답 DTO와 이메일 중복 검사 로직을 수정함. 응답 객체에서 `password`가 노출되지 않도록 `UserResponse`를 사용해 필요한 정보만 반환하도록 변경하고, `name` 필드에는 실제 사용자 이름이 들어가도록 매핑을 수정함. 또한 이메일 중복 검사는 대소문자를 구분하지 않도록 변경하여 `Test@example.com`과 `test@example.com`을 같은 이메일로 처리하도록 개선함.

## 3. 가장 중요하다고 생각한 수정

존재하지 않는 사용자와 삭제된 사용자에 대한 예외 처리의 우선도를 가장 높게 두고 수정했습니다.

기존 코드는 존재하지 않는 id를 조회, 수정, 삭제할 때 서버 내에서 예외가 발생할 가능성이 있었으며,
이는 클라이언트 입장에서 요청 실패의 원인을 명확히 알 수 없게 만들었습니다. 또한 삭제된 사용자가 여전히 조회 결과에 포함될 수 있는 문제도 있었습니다.

REST API에서는 예외로 인해 서버가 `500 Internal Server Error`를 반환하는 것보다, 
클라이언트가 요청의 실패 원인을 명확히 알 수 있도록 `404 Not Found`와 같은 적절한 상태 코드를 반환하는 것이 중요하다고 생각했습니다. 
이를 통해 클라이언트는 요청이 실패한 이유를 명확히 이해하고, 사용자에게 적절한 피드백을 제공할 수 있게 되었기 때문입니다.
